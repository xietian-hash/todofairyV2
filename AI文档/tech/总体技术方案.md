# 总体技术方案

## 1. 架构目标

将任务精灵重构为标准 Monorepo：

- `apps/api`：NestJS 后端服务
- `apps/miniprogram`：原生微信小程序
- `packages/*`：后续共享类型、常量、工具

## 2. 技术选型

- 后端：Node.js + NestJS + Prisma + MySQL
- 前端：原生微信小程序
- 契约：REST + OpenAPI
- 鉴权：JWT
- 校验：class-validator

## 3. 当前实现

### 3.1 数据层

当前后端已不再使用 `data/dev-db.json` 文件存储，统一改为 `Prisma + MySQL`：

- Prisma schema：`apps/api/prisma/schema.prisma`
- 数据连接：根目录 `.env` 中的 `DATABASE_URL`
- 初始化方式：`pnpm install` 后执行 `pnpm run prisma:push`

### 3.2 已完成模块

- `auth`：微信登录、开发态登录、JWT 签发
- `tags`：标签增删改查
- `tasks`：任务增删改查、任务统计
- `todos`：待办生成、状态切换、补偿、父子待办联动
- `calendar`：月历状态统计
- `notifications`：通知设置、测试发送、预览、日志
- `internal`：日切、日报、周报、数据库初始化信息

## 4. 关键约束

- 父任务统计口径不能回退
- 周报自动发送仍只允许周一执行
- 一次性任务创建后生成当天待办，保留原有自动归档语义
- 子待办全部完成后，父待办自动完成
- 所有用户侧资源访问必须经过 JWT 和 `userId` 归属校验

## 5. 定时任务方案

当前内部接口保留以下能力，供后续调度器接入：

- `POST /api/v1/internal/daily-rollover`
- `POST /api/v1/internal/daily-summary-notify`
- `POST /api/v1/internal/weekly-summary-notify`
- `POST /api/v1/internal/db-init`

调用时必须在请求头中传入 `x-internal-key`，并与环境变量 `INTERNAL_API_KEY` 一致。
